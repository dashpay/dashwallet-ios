# DashSync → SwiftDashSDK Migration Map

A function-first inventory of what needs to move from DashSync (Obj-C, CocoaPods) to SwiftDashSDK (Swift, SPM at `../platform/packages/swift-sdk`). Organized by user-facing or internal capability rather than by file, so the same file appears under multiple functions where relevant.

**SDK column legend:**
- 🟢 **Ready** — SwiftDashSDK exposes a working Swift API today
- 🟡 **Partial** — Rust FFI exists, Swift wrapper missing or incomplete
- 🔴 **Blocked** — Not implemented in either layer; requires upstream work
- ⚪ **N/A** — Stays on DashSync forever, or replaceable with iOS SDK primitives

**Status column legend** (where we are in the migration of each function):
- `—` Not started
- `🌓 Shadow` Adapter wired in, both libraries called, DashSync still authoritative (Stage 0)
- `🌗 Flipped` SwiftDashSDK is authoritative, DashSync still called for safety (Stage 1)
- `🌘 Solo` Only SwiftDashSDK is called from the call sites, adapter still exists (Stage 2)
- `✅ Done` Adapter retired, call sites use SwiftDashSDK directly (Stage 3)

**Effort:** S = a few hours, M = a day or two, L = a week, XL = multi-week

**Priority:** based on (a) how user-visible the function is and (b) how much it unblocks other work

## Storage migration

> **Key material migration is tracked separately** in [`DASHSYNC_KEY_MIGRATION.md`](./DASHSYNC_KEY_MIGRATION.md). That doc covers DashSync mnemonic import into the host-owned SwiftDashSDK wallet plus `WalletStorage`. This section covers the **non-key** persisted data (tx history, masternode list, identity entities, etc.).

**Earlier versions of this doc claimed a Core Data → SwiftData migrator was a hard blocker for functions #5/#6/#7/#11. After the #5 work landed, that turned out to be wrong.** The data we'd need to "migrate" splits cleanly into two buckets:

**Bucket 1: chain-derived data** (tx history, UTXOs, address used/unused state, masternode list, sync state, balance) — all of this is **re-derivable from the chain**. SwiftDashSDK's SPV client persists chain headers and scanned blocks at `Documents/SwiftDashSDK/SPV/<network>/`, so on cold launch the in-memory wallet state is repopulated by replaying scanned blocks against the imported wallet keys. There's no need to copy anything from DashSync's Core Data — SwiftDashSDK rebuilds it from first principles. Cold launch takes a few seconds (chain headers are on disk; only the wallet-specific UTXO recovery runs), not minutes.

**Bucket 2: user-entered metadata** (tx categories, tax categories, memos, gift card receipts, custom icons, address labels) — this is irreplaceable but **never lived in DashSync's Core Data in the first place**. dashwallet-ios stores all of it in its **own SQLite database** via `DatabaseConnection.shared.db`:
- `transaction_metadata` (`TransactionMetadataDAOImpl`) — keyed by `txHash`. Holds tax category, custom labels, gift card linkage.
- `address_userinfo` (`AddressUserInfo.swift`) — keyed by address. Holds tax category per address.
- Other tables for explore merchants, gift card providers, etc.

Because the linkage key is `txHash` (deterministic from on-chain bytes), the metadata stays attached to transactions automatically after SPV resync — same hash, same row.

**What this means for the migration:**

| Data | Where it lives | Action needed |
|---|---|---|
| BIP44 addresses (used / unused) | DashSync Core Data | None — re-derive via SPV scan |
| Transaction history | DashSync Core Data | None — re-derive via SPV scan |
| Masternode list cache | DashSync Core Data | None — re-fetch on first launch |
| Identity entities + DashPay contacts | DashSync Core Data | None — re-fetch via Platform queries |
| Sync state (last block height, etc) | DashSync Core Data | None — re-derive |
| **Tx categories, memos, gift card receipts** | dashwallet's own SQLite | **None — already preserved, independent of DashSync** |
| **Address labels, tax categories** | dashwallet's own SQLite | **None — already preserved, independent of DashSync** |

**Net result: no Core Data → SwiftData migrator is required.** The "migrator" entry was removed from the Hard Blockers list. The only remaining storage concern is the **per-network SPV chain data dir** at `Documents/SwiftDashSDK/SPV/<network>/`, which SwiftDashSDK creates and manages itself — no migration needed there either.

The one hypothetical caveat: any DashPay contact request data that's in DashSync's Core Data and not in dashwallet's own SQLite would be re-fetched from Platform on first launch after #18 lands. That's not a "migration" — it's a one-time refresh.

## Known SDK issues (filed against `platform/packages/swift-sdk`)

Bugs discovered during dashwallet integration that block specific paths. Tracked here so they're not lost between sessions — dashwallet-side code is correct and waiting on upstream fixes.

### 1. ~~DashPay profile write fails with "could not decode data contracts query"~~ — RESOLVED 2026-05-27

**Resolved by:** Pinning Platform protocol version 11 at SDK init (`SwiftDashSDKHost.swift` — `SDK(network:platformVersion: 11)`). The auto-detect default was selecting a newer protocol version whose `GetDataContractsRequest.version = None` shape the server rejects. Upstream PR [dashpay/platform#3751](https://github.com/dashpay/platform/pull/3751) (`feat/rs-sdk-ffi-config-version`, dashpay-org-canonical) adds a `platform_version: u32` field to `DashSDKConfig` (ABI-stable tail append, `0` = auto-detect). The earlier fork PR #3734 was superseded by this approach — single config field instead of duplicate `_with_protocol_version` FFI functions.

**End-to-end verified 2026-05-27 on testnet:**
- `wallet.updateDashPayProfile` succeeds via the "create-then-update wrong-verb fallback" path; subsequent `getDashPayProfile` reads return the freshly-saved fields; the post-save snapshot refresh updates the UI immediately.
- Same fix also unblocks the queries Row #18a's status view originally needed (`wallet.syncDpnsNames`, `wallet.fetchContestVoteState`) — verified: `syncDpnsNames` now returns the full DPNS name list for the identity instead of the previous gRPC decode error.

**Original symptom (kept for archeology):** Both `createDashPayProfile` and `updateDashPayProfile` failed with `decoding error: could not decode data contracts query`, originating server-side at `rs-drive-abci/src/query/data_contract_based_queries/data_contracts/mod.rs:22-24` when `GetDataContractsRequest.version` was `None`. DPNS registration and identity registration weren't affected because they took different request paths.

**Follow-up:** When PR #3751 lands on a tagged `platform` release, drop the explicit `platformVersion: 11` argument and rely on the auto-detect default again.
