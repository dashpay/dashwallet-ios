# DashSync → SwiftDashSDK Migration Map

A function-first inventory of what needs to move from DashSync (Obj-C, CocoaPods) to SwiftDashSDK (Swift, SPM at `../platform/packages/swift-sdk`). Organized by user-facing or internal capability rather than by file, so the same file appears under multiple functions where relevant.

**SDK column legend:**
- 🟢 **Ready** — SwiftDashSDK exposes a working Swift API today
- 🟡 **Partial** — Rust FFI exists, Swift wrapper missing or incomplete
- 🔴 **Blocked** — Not implemented in either layer; requires upstream work
- ⚪ **N/A** — Stays on DashSync forever, or replaceable with iOS SDK primitives

**Status column legend** (where we are in the migration of each function):
- `—` Not started
- `🌓 Shadow` Adapter wired in, both libraries called, DashSync still authoritative (Stage 0)
- `🌗 Flipped` SwiftDashSDK is authoritative, DashSync still called for safety (Stage 1)
- `🌘 Solo` Only SwiftDashSDK is called from the call sites, adapter still exists (Stage 2)
- `✅ Done` Adapter retired, call sites use SwiftDashSDK directly (Stage 3)

**Effort:** S = a few hours, M = a day or two, L = a week, XL = multi-week

**Priority:** based on (a) how user-visible the function is and (b) how much it unblocks other work

| **#14 Wipe wallet** | 🌗 Flipped | DashSync's wipe flow now also clears SwiftDashSDK state via `DWSwiftDashSDKWalletWiper`, a standalone Swift observer on `DWWillWipeWalletNotification` registered at app launch. All user-facing wipe entry points are covered through that single notification because they all funnel through `[DWEnvironment clearAllWalletsAndRemovePin:]`. The wiper deletes the encrypted seed and mnemonic via `WalletStorage().deleteSeed()` / `deleteMnemonic()`, deletes **both** supported-network runtime descriptors (`wallet.descriptor.mainnet`, `wallet.descriptor.testnet`), and then calls `SwiftDashSDKWalletRuntime.handleWalletWiped()` so the app-owned runtime performs the rest of the teardown in one place: stop/reset the SPV coordinator, invalidate the detached-wallet cache, and clear published balance/transaction state. The chain data dir at `Documents/SwiftDashSDK/SPV/<network>/` is intentionally NOT deleted — chain data is public, and leaving it in place lets the next wallet on the same device skip an expensive resync. Logger category: `swift-sdk-migration.wallet-wiper`. |
| **#10 PIN auth (PIN-change half)** | 🌗 Flipped | DashSync's PIN-change flow now also re-encrypts the SwiftDashSDK seed via `DWSwiftDashSDKPinChanger`, called from `DWSetPinModel.setPin:` immediately after `[DSAuthenticationManager setupNewPin:]` succeeds. The model captures the current PIN via `[DSAuthenticationManager getPin:]` BEFORE the DashSync call (which would otherwise overwrite it via `setKeychainString(pin, PIN_KEY, NO)` at `DSAuthenticationManager.m:751-758`), then dispatches the SwiftDashSDK re-encryption to a background queue. The mirror retrieves the existing seed with the old PIN, calls `WalletStorage.storeSeed(seed, pin: newPin)` (which is already idempotent — deletes existing entry first per `WalletStorage.swift:41`), and round-trip verifies with the new PIN. **Self-discriminates first-time PIN setup** (no `oldPin` captured because `DSAuthenticationManager hasPin:` is false at that point → mirror call skipped — the wallet creator will populate the seed later in the onboarding flow with the new PIN) and **no-op changes** where old == new. Three error branches: `seedNotFound` no-ops with an info log (defensive — should never fire on a working wallet), `invalidPIN` deletes the orphaned seed and logs a warning (indicates pre-existing drift from before this PR shipped), any other error best-effort deletes to avoid leaving a half-written entry. **No DashSync code paths were modified.** Logger category: `swift-sdk-migration.pin-changer`. |
| **#5 Wallet balance** | 🌗 Flipped | Three commits unfreeze the home screen number and the surrounding balance consumers post-M6. That one belongs to a future chain-state commit. **Important:** the original "Recommended order" had #5 waiting on a Core Data → SwiftData storage migrator. That migrator turns out to be unnecessary — wallet UTXO state lives in SwiftDashSDK's in-memory FFI state and is repopulated from on-disk chain block data at `Documents/SwiftDashSDK/SPV/<network>/` on cold launch; user-added tx metadata (categories, memos, gift card receipts) lives in dashwallet's own SQLite (`TransactionMetadataDAOImpl` keyed by `txHash`), entirely independent of DashSync's Core Data. See "Storage migration" section for the corrected picture. Logger category: `swift-sdk-migration.wallet-state` (`💰 WALLET ::`). |
| **#11 SPV chain sync (M5 + M6)** | 🌗 Flipped | M5 (`3cf5962e5`) rewrote `SyncingActivityMonitor` to source sync state from `SwiftDashSDKSPVCoordinator` via a `combineLatest` subscription on `$state, $progress, $syncProgress, $bestPeerHeight` instead of DashSync's `DSChainManagerSync*` notifications. Public API surface remains unchanged (`progress`, `state`, `forceStartSyncingActivity`, observer protocol, `.syncStateChangedNotification`) so existing consumers still work. The key follow-up is runtime ownership: app launch, manual restart, wallet create/import/migrate, wipe, and network changes now go through `DWSwiftDashSDKWalletRuntime`, which performs an ordered stop/reset/invalidate/clear/restore/start sequence and hands an explicit detached `HDWallet` into the lower-level coordinator. `SwiftDashSDKSPVCoordinator` no longer polls `DWEnvironment`, `UserDefaults`, or the wallet provider on its own startup path; it just owns `SPVClient` lifecycle plus published sync state. Background fetch is still rewired to return `UIBackgroundFetchResultNoData`. DashSync's `setupDashSyncOnce` remains in place because masternode list infrastructure and a handful of read APIs still depend on it; the chain manager exists but no longer syncs because nothing calls `startSync` anymore. Storage migration is still unnecessary — SwiftDashSDK's chain headers + scanned-block cache live at `Documents/SwiftDashSDK/SPV/<network>/` and repopulate from peers on cold launch. Logger category: `swift-sdk-migration.spv-coordinator` (`🛰️ SPVCOORD ::`). |
| **#12 Network switch** | 🌗 Flipped | Mainnet/testnet runtime switching is now centralized in `DWSwiftDashSDKWalletRuntime`. The runtime observes `DWCurrentNetworkDidChangeNotification`, maps `DWEnvironment.currentChain` explicitly to `.mainnet` / `.testnet`, and treats everything else as unsupported instead of silently falling back. On launch and every supported-network change it runs the same queue-serialized sequence: stop the coordinator with a full runtime reset, invalidate the detached-wallet cache, clear published balance/transactions, ensure a descriptor exists for the selected network, restore that network's detached `HDWallet`, and start the coordinator with it. Runtime descriptors are stored per network in app-owned Keychain accounts (`wallet.descriptor.mainnet`, `wallet.descriptor.testnet`), so switching reuses the correct descriptor deterministically and can bootstrap a missing descriptor once from mnemonic material when the chain already has a wallet. Unsupported networks (devnet/regtest/legacy Evonet-style chains) stop the runtime, clear state, and surface an explicit runtime error instead of falling back to testnet. |
| **#13 Backup seed phrase** | 🌗 Flipped | **First read-side flip in the migration arc.** The Settings → View Recovery Phrase path now reads the mnemonic from SwiftDashSDK's `WalletStorage.retrieveMnemonic()` via `DWSwiftDashSDKMnemonicReader`, with no DashSync fallback. Upstream `WalletStorage.swift` gained three new methods (`storeMnemonic`/`retrieveMnemonic`/`deleteMnemonic`) — plain keychain at `wallet.mnemonic`, no PIN encryption (iOS keychain is the security boundary). All write-side mirrors updated: `SwiftDashSDKWalletCreator` stores the mnemonic alongside the seed during create/import, `SwiftDashSDKKeyMigrator` stores it during the one-shot upgrade migration, `SwiftDashSDKWalletWiper` deletes it during wipe. A new `SwiftDashSDKMnemonicBackfiller` runs once at app launch to copy the mnemonic from DashSync's keychain (`WALLET_MNEMONIC_KEY_<uniqueID>` at `DSWallet.m:67,464`) for users who ran the migrator/creator before this PR shipped. The onboarding path (just-created wallet) now also reads from SwiftDashSDK after the #3 mnemonic generation flip shipped. **No DashSync code paths were modified.** `SwiftDashSDKPinChanger` was NOT updated — since the mnemonic isn't PIN-encrypted, PIN changes don't affect it. |

| **#8 Send Dash** | 🌗 Flipped | Standard sends now go through one shared app boundary, `WalletSendService`, instead of letting `DWPaymentProcessor` and `SendCoinsService` each own their own send stack. ## Storage migration

> **Key material migration is tracked separately** in [`DASHSYNC_KEY_MIGRATION.md`](./DASHSYNC_KEY_MIGRATION.md). That doc covers mnemonic → seed → `WalletStorage` plus the app-owned runtime wallet descriptor in Keychain, and the per-PR status of the multi-phase key migration arc. This section covers the **non-key** persisted data (tx history, masternode list, identity entities, etc.).

**Earlier versions of this doc claimed a Core Data → SwiftData migrator was a hard blocker for functions #5/#6/#7/#11. After the #5 work landed, that turned out to be wrong.** The data we'd need to "migrate" splits cleanly into two buckets:

**Bucket 1: chain-derived data** (tx history, UTXOs, address used/unused state, masternode list, sync state, balance) — all of this is **re-derivable from the chain**. SwiftDashSDK's SPV client persists chain headers and scanned blocks at `Documents/SwiftDashSDK/SPV/<network>/`, so on cold launch the in-memory wallet state is repopulated by replaying scanned blocks against the imported wallet keys. There's no need to copy anything from DashSync's Core Data — SwiftDashSDK rebuilds it from first principles. Cold launch takes a few seconds (chain headers are on disk; only the wallet-specific UTXO recovery runs), not minutes.

**Bucket 2: user-entered metadata** (tx categories, tax categories, memos, gift card receipts, custom icons, address labels) — this is irreplaceable but **never lived in DashSync's Core Data in the first place**. dashwallet-ios stores all of it in its **own SQLite database** via `DatabaseConnection.shared.db`:
- `transaction_metadata` (`TransactionMetadataDAOImpl`) — keyed by `txHash`. Holds tax category, custom labels, gift card linkage.
- `address_userinfo` (`AddressUserInfo.swift`) — keyed by address. Holds tax category per address.
- Other tables for explore merchants, gift card providers, etc.

Because the linkage key is `txHash` (deterministic from on-chain bytes), the metadata stays attached to transactions automatically after SPV resync — same hash, same row.

**What this means for the migration:**

| Data | Where it lives | Action needed |
|---|---|---|
| BIP44 addresses (used / unused) | DashSync Core Data | None — re-derive via SPV scan |
| Transaction history | DashSync Core Data | None — re-derive via SPV scan |
| Masternode list cache | DashSync Core Data | None — re-fetch on first launch |
| Identity entities + DashPay contacts | DashSync Core Data | None — re-fetch via Platform queries |
| Sync state (last block height, etc) | DashSync Core Data | None — re-derive |
| **Tx categories, memos, gift card receipts** | dashwallet's own SQLite | **None — already preserved, independent of DashSync** |
| **Address labels, tax categories** | dashwallet's own SQLite | **None — already preserved, independent of DashSync** |

**Net result: no Core Data → SwiftData migrator is required.** The "migrator" entry was removed from the Hard Blockers list. The only remaining storage concern is the **per-network SPV chain data dir** at `Documents/SwiftDashSDK/SPV/<network>/`, which SwiftDashSDK creates and manages itself — no migration needed there either.

The one hypothetical caveat: any DashPay contact request data that's in DashSync's Core Data and not in dashwallet's own SQLite would be re-fetched from Platform on first launch after #18 lands. That's not a "migration" — it's a one-time refresh.

## Hard blockers (do these first or migration is dead in the water)

1. **Send transaction (#8)** — wrap `wallet_build_and_sign_transaction()` and `dash_spv_ffi_client_broadcast_transaction()` in Swift. Lives in `platform` repo, not here.
2. **Identity FFI signature mismatches (#16)** — 6+ functions have parameter type mismatches between Swift wrappers and the C signatures. Lives in `platform` repo.
3. **DashPay contact service (#18)** — currently a stub. Lives in `platform` repo.

~~Core Data → SwiftData migrator~~ — was previously listed as a hard blocker. Removed after the #5 work landed (see "Storage migration" section for the corrected analysis).

CoinJoin (#20) is **not** a hard blocker because we can keep DashSync linked solely for that path during the transition (or drop the feature). Don't let it stop everything else.

## Notes

- **The two libraries can coexist.** SwiftDashSDK is already linked alongside DashSync in the `dashwallet` target (as of commit `8309e1ef4`). Migration happens function-by-function; both libraries stay linked until Wave 5 completes.
- **`DWEnvironment`** is the singleton that fans out to almost every function above. The cleanest migration shape is to refactor it into a thin facade with two backends (`DashSyncBackend` and `SwiftDashSDKBackend`) selected by feature flag, so individual functions can flip independently without one big-bang cutover.
