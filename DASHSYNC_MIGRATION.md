# DashSync тЖТ SwiftDashSDK Migration Map

A function-first inventory of what needs to move from DashSync (Obj-C, CocoaPods) to SwiftDashSDK (Swift, SPM at `../platform/packages/swift-sdk`). Organized by user-facing or internal capability rather than by file, so the same file appears under multiple functions where relevant.

**SDK column legend:**
- ЁЯЯв **Ready** тАФ SwiftDashSDK exposes a working Swift API today
- ЁЯЯб **Partial** тАФ Rust FFI exists, Swift wrapper missing or incomplete
- ЁЯФ┤ **Blocked** тАФ Not implemented in either layer; requires upstream work
- тЪк **N/A** тАФ Stays on DashSync forever, or replaceable with iOS SDK primitives

**Status column legend** (where we are in the migration of each function):
- `тАФ` Not started
- `ЁЯМУ Shadow` Adapter wired in, both libraries called, DashSync still authoritative (Stage 0)
- `ЁЯМЧ Flipped` SwiftDashSDK is authoritative, DashSync still called for safety (Stage 1)
- `ЁЯМШ Solo` Only SwiftDashSDK is called from the call sites, adapter still exists (Stage 2)
- `тЬЕ Done` Adapter retired, call sites use SwiftDashSDK directly (Stage 3)

**Effort:** S = a few hours, M = a day or two, L = a week, XL = multi-week

**Priority:** based on (a) how user-visible the function is and (b) how much it unblocks other work

## Storage migration

> **Key material migration is tracked separately** in [`DASHSYNC_KEY_MIGRATION.md`](./DASHSYNC_KEY_MIGRATION.md). That doc covers mnemonic тЖТ seed тЖТ `WalletStorage` + `HDWallet` and the per-PR status of the multi-phase key migration arc. This section covers the **non-key** persisted data (tx history, masternode list, identity entities, etc.).

**Earlier versions of this doc claimed a Core Data тЖТ SwiftData migrator was a hard blocker for functions #5/#6/#7/#11. After the #5 work landed, that turned out to be wrong.** The data we'd need to "migrate" splits cleanly into two buckets:

**Bucket 1: chain-derived data** (tx history, UTXOs, address used/unused state, masternode list, sync state, balance) тАФ all of this is **re-derivable from the chain**. SwiftDashSDK's SPV client persists chain headers and scanned blocks at `Documents/SwiftDashSDK/SPV/<network>/`, so on cold launch the in-memory wallet state is repopulated by replaying scanned blocks against the imported wallet keys. There's no need to copy anything from DashSync's Core Data тАФ SwiftDashSDK rebuilds it from first principles. Cold launch takes a few seconds (chain headers are on disk; only the wallet-specific UTXO recovery runs), not minutes.

**Bucket 2: user-entered metadata** (tx categories, tax categories, memos, gift card receipts, custom icons, address labels) тАФ this is irreplaceable but **never lived in DashSync's Core Data in the first place**. dashwallet-ios stores all of it in its **own SQLite database** via `DatabaseConnection.shared.db`:
- `transaction_metadata` (`TransactionMetadataDAOImpl`) тАФ keyed by `txHash`. Holds tax category, custom labels, gift card linkage.
- `address_userinfo` (`AddressUserInfo.swift`) тАФ keyed by address. Holds tax category per address.
- Other tables for explore merchants, gift card providers, etc.

Because the linkage key is `txHash` (deterministic from on-chain bytes), the metadata stays attached to transactions automatically after SPV resync тАФ same hash, same row.

**What this means for the migration:**

| Data | Where it lives | Action needed |
|---|---|---|
| BIP44 addresses (used / unused) | DashSync Core Data | None тАФ re-derive via SPV scan |
| Transaction history | DashSync Core Data | None тАФ re-derive via SPV scan |
| Masternode list cache | DashSync Core Data | None тАФ re-fetch on first launch |
| Identity entities + DashPay contacts | DashSync Core Data | None тАФ re-fetch via Platform queries |
| Sync state (last block height, etc) | DashSync Core Data | None тАФ re-derive |
| **Tx categories, memos, gift card receipts** | dashwallet's own SQLite | **None тАФ already preserved, independent of DashSync** |
| **Address labels, tax categories** | dashwallet's own SQLite | **None тАФ already preserved, independent of DashSync** |

**Net result: no Core Data тЖТ SwiftData migrator is required.** The "migrator" entry was removed from the Hard Blockers list. The only remaining storage concern is the **per-network SPV chain data dir** at `Documents/SwiftDashSDK/SPV/<network>/`, which SwiftDashSDK creates and manages itself тАФ no migration needed there either.

The one hypothetical caveat: any DashPay contact request data that's in DashSync's Core Data and not in dashwallet's own SQLite would be re-fetched from Platform on first launch after #18 lands. That's not a "migration" тАФ it's a one-time refresh.

## Notes

- **The two libraries can coexist.** SwiftDashSDK is already linked alongside DashSync in the `dashwallet` target (as of commit `8309e1ef4`). Migration happens function-by-function; both libraries stay linked until Wave 5 completes.
- **`DWEnvironment`** is the singleton that fans out to almost every function above. The cleanest migration shape is to refactor it into a thin facade with two backends (`DashSyncBackend` and `SwiftDashSDKBackend`) selected by feature flag, so individual functions can flip independently without one big-bang cutover.
