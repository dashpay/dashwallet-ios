# DashSync тЖТ SwiftDashSDK Migration Map

A function-first inventory of what needs to move from DashSync (Obj-C, CocoaPods) to SwiftDashSDK (Swift, SPM at `../platform/packages/swift-sdk`). Organized by user-facing or internal capability rather than by file, so the same file appears under multiple functions where relevant.

**SDK column legend:**
- ЁЯЯв **Ready** тАФ SwiftDashSDK exposes a working Swift API today
- ЁЯЯб **Partial** тАФ Rust FFI exists, Swift wrapper missing or incomplete
- ЁЯФ┤ **Blocked** тАФ Not implemented in either layer; requires upstream work
- тЪк **N/A** тАФ Stays on DashSync forever, or replaceable with iOS SDK primitives

**Status column legend** (where we are in the migration of each function):
- `тАФ` Not started
- `ЁЯМУ Shadow` Adapter wired in, both libraries called, DashSync still authoritative (Stage 0)
- `ЁЯМЧ Flipped` SwiftDashSDK is authoritative, DashSync still called for safety (Stage 1)
- `ЁЯМШ Solo` Only SwiftDashSDK is called from the call sites, adapter still exists (Stage 2)
- `тЬЕ Done` Adapter retired, call sites use SwiftDashSDK directly (Stage 3)

**Effort:** S = a few hours, M = a day or two, L = a week, XL = multi-week

**Priority:** based on (a) how user-visible the function is and (b) how much it unblocks other work

| **#14 Wipe wallet** | ЁЯМЧ Flipped | DashSync's wipe flow now also clears SwiftDashSDK state via `DWSwiftDashSDKWalletWiper`, a standalone Swift observer on `DWWillWipeWalletNotification` registered at app launch. The chain data dir is intentionally NOT deleted тАФ chain data is public, and leaving it in place lets the next wallet on the same device skip an expensive resync. | **#10 PIN auth (PIN-change half)** | ЁЯМЧ Flipped | PIN-change mirroring into SwiftDashSDK is no longer required. The `DWSwiftDashSDKPinChanger` adapter and its invocation from `DWSetPinModel.setPin:` have been deleted; `setPin:` is now a thin wrapper around `[DSAuthenticationManager setupNewPin:]`. **No DashSync code paths were modified** тАФ PIN handling remains DashSync's responsibility. |
| **#5 Wallet balance** | ЁЯМЧ Flipped | Three commits unfreeze the home screen number and the surrounding balance consumers post-M6. The Apple Watch sync-gate in `DWPhoneWCSessionManager` was also fixed in commit 3 тАФ it had been reading `currentChainManager.combinedSyncProgress == 1.0` (DashSync, frozen post-M6) and now reads `[SyncingActivityMonitor shared].state == SyncingActivityMonitorStateSyncDone` (live SwiftDashSDK sync state via M5's monitor). **Still on DashSync:** CrowdNode / CoinJoin / Coinbase balance reads тАФ they use different APIs entirely (`account.maxOutputAmount`, `coinJoinBalance`, exchange-side balances) and need their own follow-ups. `DWRecoverModel.isWalletEmpty` is also still on DashSync but for a different reason тАФ it reads BOTH balance AND chain state (`lastSyncBlockTimestamp`, `lastSyncBlockHeight`), and chain state is also frozen post-M6, so a half-fix would create false-positive "wallet not empty" results during recovery. That one belongs to a future chain-state commit. **Important:** the original "Recommended order" had #5 waiting on a Core Data тЖТ SwiftData storage migrator. That migrator turns out to be unnecessary тАФ wallet UTXO state lives in SwiftDashSDK's in-memory FFI state and is repopulated from on-disk chain block data at `Documents/SwiftDashSDK/SPV/<network>/` on cold launch; user-added tx metadata (categories, memos, gift card receipts) lives in dashwallet's own SQLite (`TransactionMetadataDAOImpl` keyed by `txHash`), entirely independent of DashSync's Core Data. See "Storage migration" section for the corrected picture. Logger category: `swift-sdk-migration.wallet-state` (`ЁЯТ░ WALLET ::`). |
| **#11 SPV chain sync (M5 + M6)** | ЁЯМЧ Flipped | M5 (`3cf5962e5`) rewrote `SyncingActivityMonitor` to source sync state from `SwiftDashSDKSPVCoordinator` via a `combineLatest` subscription on `$state, $progress, $syncProgress, $bestPeerHeight` instead of DashSync's `DSChainManagerSync*` notifications. Public API surface remains unchanged (`progress`, `state`, `forceStartSyncingActivity`, observer protocol, `.syncStateChangedNotification`) so existing consumers still work. Runtime ownership now flows through `SwiftDashSDKHost`: app launch, manual restart, wallet create/import/migrate, wipe, and network changes share one `PlatformWalletManager` and one `ManagedPlatformWallet` across Core SPV and BLAST. `SwiftDashSDKSPVCoordinator` starts by network and resolves the live manager/wallet from the host; it no longer restores an app-owned wallet carrier. Background fetch is still rewired to return `UIBackgroundFetchResultNoData`. DashSync's `setupDashSyncOnce` remains in place because masternode list infrastructure and a handful of read APIs still depend on it; the chain manager exists but no longer syncs because nothing calls `startSync` anymore. Storage migration is still unnecessary тАФ SwiftDashSDK's chain headers + scanned-block cache live at `Documents/SwiftDashSDK/SPV/<network>/` and repopulate from peers on cold launch. Logger category: `swift-sdk-migration.spv-coordinator` (`ЁЯЫ░я╕П SPVCOORD ::`). |
| **#12 Network switch** | ЁЯМЧ Flipped | Mainnet/testnet runtime switching is now centralized in `DWSwiftDashSDKWalletRuntime`. The runtime observes `DWCurrentNetworkDidChangeNotification`, maps `DWEnvironment.currentChain` explicitly to `.mainnet` / `.testnet`, and treats everything else as unsupported instead of silently falling back. On launch and every supported-network change it runs the same queue-serialized sequence: stop BLAST/SPV, clear published wallet state, ask `SwiftDashSDKHost` to load the selected network's persisted `ManagedPlatformWallet`, and start SPV/BLAST from that shared host state. Unsupported networks (devnet/regtest/legacy Evonet-style chains) stop the runtime, clear state, and surface an explicit runtime error instead of falling back to testnet. |
| **#13 Backup seed phrase** | ЁЯМЧ Flipped | **First read-side flip in the migration arc.** The Settings тЖТ View Recovery Phrase path now reads the mnemonic from SwiftDashSDK's `WalletStorage.retrieveMnemonic()` via `[self readStoredMnemonic]` тАФ a co-located `@objc` Swift extension on `DWPreviewSeedPhraseModel` (`DWPreviewSeedPhraseModel+Mnemonic.swift`) тАФ with no DashSync fallback. Upstream `WalletStorage.swift` gained three new methods (`storeMnemonic`/`retrieveMnemonic`/`deleteMnemonic`) тАФ plain keychain at `wallet.mnemonic`, no PIN encryption (iOS keychain is the security boundary). All write-side mirrors updated: `SwiftDashSDKWalletCreator` stores the mnemonic alongside the seed during create/import, `SwiftDashSDKKeyMigrator` stores it during the one-shot upgrade migration, `SwiftDashSDKWalletWiper` deletes it during wipe. The onboarding path (just-created wallet) reads from SwiftDashSDK through the same extension. The standalone `DWSwiftDashSDKMnemonicReader` adapter class was retired in favor of the co-located extension. **No DashSync code paths were modified.** Because the mnemonic isn't PIN-encrypted, PIN changes don't affect SwiftDashSDK storage at all. |

## Storage migration

> **Key material migration is tracked separately** in [`DASHSYNC_KEY_MIGRATION.md`](./DASHSYNC_KEY_MIGRATION.md). That doc covers DashSync mnemonic import into the host-owned SwiftDashSDK wallet plus `WalletStorage`. This section covers the **non-key** persisted data (tx history, masternode list, identity entities, etc.).

**Earlier versions of this doc claimed a Core Data тЖТ SwiftData migrator was a hard blocker for functions #5/#6/#7/#11. After the #5 work landed, that turned out to be wrong.** The data we'd need to "migrate" splits cleanly into two buckets:

**Bucket 1: chain-derived data** (tx history, UTXOs, address used/unused state, masternode list, sync state, balance) тАФ all of this is **re-derivable from the chain**. SwiftDashSDK's SPV client persists chain headers and scanned blocks at `Documents/SwiftDashSDK/SPV/<network>/`, so on cold launch the in-memory wallet state is repopulated by replaying scanned blocks against the imported wallet keys. There's no need to copy anything from DashSync's Core Data тАФ SwiftDashSDK rebuilds it from first principles. Cold launch takes a few seconds (chain headers are on disk; only the wallet-specific UTXO recovery runs), not minutes.

**Bucket 2: user-entered metadata** (tx categories, tax categories, memos, gift card receipts, custom icons, address labels) тАФ this is irreplaceable but **never lived in DashSync's Core Data in the first place**. dashwallet-ios stores all of it in its **own SQLite database** via `DatabaseConnection.shared.db`:
- `transaction_metadata` (`TransactionMetadataDAOImpl`) тАФ keyed by `txHash`. Holds tax category, custom labels, gift card linkage.
- `address_userinfo` (`AddressUserInfo.swift`) тАФ keyed by address. Holds tax category per address.
- Other tables for explore merchants, gift card providers, etc.

Because the linkage key is `txHash` (deterministic from on-chain bytes), the metadata stays attached to transactions automatically after SPV resync тАФ same hash, same row.

**What this means for the migration:**

| Data | Where it lives | Action needed |
|---|---|---|
| BIP44 addresses (used / unused) | DashSync Core Data | None тАФ re-derive via SPV scan |
| Transaction history | DashSync Core Data | None тАФ re-derive via SPV scan |
| Masternode list cache | DashSync Core Data | None тАФ re-fetch on first launch |
| Identity entities + DashPay contacts | DashSync Core Data | None тАФ re-fetch via Platform queries |
| Sync state (last block height, etc) | DashSync Core Data | None тАФ re-derive |
| **Tx categories, memos, gift card receipts** | dashwallet's own SQLite | **None тАФ already preserved, independent of DashSync** |
| **Address labels, tax categories** | dashwallet's own SQLite | **None тАФ already preserved, independent of DashSync** |

**Net result: no Core Data тЖТ SwiftData migrator is required.** The "migrator" entry was removed from the Hard Blockers list. The only remaining storage concern is the **per-network SPV chain data dir** at `Documents/SwiftDashSDK/SPV/<network>/`, which SwiftDashSDK creates and manages itself тАФ no migration needed there either.

The one hypothetical caveat: any DashPay contact request data that's in DashSync's Core Data and not in dashwallet's own SQLite would be re-fetched from Platform on first launch after #18 lands. That's not a "migration" тАФ it's a one-time refresh.

## Hard blockers (do these first or migration is dead in the water)

1. **Send transaction (#8)** тАФ wrap `wallet_build_and_sign_transaction()` and `dash_spv_ffi_client_broadcast_transaction()` in Swift. Lives in `platform` repo, not here.
2. **Identity FFI signature mismatches (#16)** тАФ 6+ functions have parameter type mismatches between Swift wrappers and the C signatures. Lives in `platform` repo.
3. **DashPay contact service (#18)** тАФ currently a stub. Lives in `platform` repo.

~~Core Data тЖТ SwiftData migrator~~ тАФ was previously listed as a hard blocker. Removed after the #5 work landed (see "Storage migration" section for the corrected analysis).

CoinJoin (#20) is **not** a hard blocker because we can keep DashSync linked solely for that path during the transition (or drop the feature). Don't let it stop everything else.

## Notes

- **The two libraries can coexist.** SwiftDashSDK is already linked alongside DashSync in the `dashwallet` target (as of commit `8309e1ef4`). Migration happens function-by-function; both libraries stay linked until Wave 5 completes.
- **`DWEnvironment`** is the singleton that fans out to almost every function above. The cleanest migration shape is to refactor it into a thin facade with two backends (`DashSyncBackend` and `SwiftDashSDKBackend`) selected by feature flag, so individual functions can flip independently without one big-bang cutover.
